## 1. Что такое ветка и зачем она нужна

Представьте репозиторий как общий документ, в котором работают все студенты одновременно. Если все будут писать в одно место — возникнет путаница. **Ветка** — это ваша личная копия репозитория, изолированная от остальных. Вы работаете в своей ветке, не мешая другим.

В этом курсе каждый студент создаёт свою ветку по шаблону `{ваш_логин}/{номер_дз}`, например `alice/hw2`. Когда работа готова — открывается **Pull Request (PR)**: предложение добавить вашу ветку в общий репозиторий. PR служит площадкой для обсуждения и рецензирования.

> ⚠️ **Важно:** пушить напрямую в ветку `main` запрещено. Вся работа ведётся только через ветки и PR.

## 2. Подготовка: установка Git и авторизация

### Установка Git

- **Windows:** скачайте с [git-scm.com](https://git-scm.com/download/win), установите с настройками по умолчанию
- **macOS:** выполните в терминале: `xcode-select —install` или `brew install git`
- **Linux:** `sudo apt install git` (Ubuntu/Debian) или `sudo dnf install git` (Fedora)

Проверьте установку:
```bash
git --version
```

### Установка GitHub CLI

GitHub CLI (`gh`) упрощает работу с GitHub из терминала.

- **Windows:** скачайте с [cli.github.com](https://cli.github.com)
- **macOS:** `brew install gh`
- **Linux:** инструкции на [cli.github.com](https://cli.github.com)

### Авторизация

```bash
gh auth login
```

Выберите:
- `GitHub.com`
- `HTTPS`
- `Login with a web browser`

Следуйте инструкциям в браузере.

### Настройка имени и email

```bash
git config --global user.name "Ваше Имя"
git config --global user.email "ваш@email.com"
```

## 3. Как сдать работу на рецензию

### Шаг 1. Клонируйте репозиторий

> ⚠️ **Не создавайте форк репозитория.** Клонируйте `peer-review-hub` напрямую и пушьте ветку в основной репозиторий, как показано ниже.

```bash
git clone https://github.com/<org>/peer-review-hub
cd peer-review-hub
```

Клонировать нужно только один раз. Для следующих ДЗ просто перейдите в папку репозитория и продолжайте с шага 2.

### Шаг 2. Обновите локальную копию

Перед началом работы всегда обновляйте репозиторий:

```bash
git checkout main
git pull origin main
```

### Шаг 3. Создайте ветку

Замените `alice` на **ваш GitHub логин**, `hw2` на номер текущего ДЗ:

```bash
git checkout -b alice/hw2
```

Эта команда создаёт новую ветку и переключается на неё.

### Шаг 4. Подготовьте файл с теорией

Создайте папку и файл (замените `alice` на ваш логин):

```bash
mkdir -p reviews/alice
```

Скопируйте теоретическую часть вашего ДЗ в файл `reviews/alice/theory.md`.

Формулы оформляйте в LaTeX. Inline: `$f(x) = x^2$`. Блочные:

```
$$
\frac{\partial L}{\partial w} = \sum_{i=1}^{n} \delta_i x_i
$$
```

### Шаг 5. Запушьте ветку

```bash
git add reviews/alice/theory.md
git commit -m "hw2: theory"
git push origin alice/hw2
```

> ⚠️ Обратите внимание: последняя команда — `git push origin alice/hw2`, **не** `git push origin main`.

### Шаг 6. Откройте Pull Request

1. Перейдите в репозиторий `peer-review-hub` на GitHub
2. Нажмите **Compare & pull request** (появится жёлтая плашка после пуша)
3. Убедитесь:
   - **base:** `main`
   - **compare:** `alice/hw2` (ваша ветка)
   - **Title:** `hw2: alice` (строго этот формат, ваш логин)
4. Заполните чеклист в описании PR
5. Нажмите **Create pull request**

> ⚠️ Не открывайте PR раньше, чем закончите теоретическую часть — после открытия PR рецензент назначается автоматически.

### Шаг 7. Дождитесь рецензента

В течение нескольких минут бот оставит комментарий с именем назначенного рецензента. Если сдавших пока мало, второй рецензент будет назначен позже — вы получите уведомление от GitHub.

## 4. Как сделать рецензию

Вам придёт уведомление от GitHub когда вас назначат рецензентом.

### Шаг 1. Откройте PR

Перейдите по ссылке из уведомления или найдите PR в репозитории `peer-review-hub`.

### Шаг 2. Изучите работу

Перейдите на вкладку **Files changed** — там виден файл `theory.md` автора.

### Шаг 3. Оставьте inline-комментарии

Наведите курсор на строку → нажмите **+** слева → напишите комментарий → **Start a review**.

Требования к комментариям:
- **Минимум 2 inline-комментария** к конкретным строкам файла
- Комментарии должны быть содержательными: указывайте на ошибки, неточности, или задавайте уточняющие вопросы
- Если всё верно — напишите почему вы согласны с формулировкой

Примеры хороших комментариев:
> «Формула для KL-дивергенции верна, но не хватает пояснения, почему результат всегда ≥ 0»

> «Здесь перепутаны индексы i и j в сумме — проверьте вывод»

> «Объяснение reparameterization trick корректное и понятное»

### Шаг 4. Завершите review

1. Нажмите кнопку **Review changes** (верхний правый угол вкладки **Files changed**)
2. Напишите общий итог рецензии (1–3 предложения)
3. Выберите один из вариантов:
   - **Approve** — работа верна, замечаний нет или они незначительные
   - **Request changes** — есть ошибки, которые нужно исправить
4. Нажмите **Submit review**

> ⚠️ Рецензия не засчитывается если вы только написали комментарии, но не нажали **Submit review**.

## 5. Требования для получения оценки

Peer review засчитывается автоматически при выполнении всех условий:

| Условие | Как проверяется |
|---|---|
| Вы сдали теоретическую часть (открыли PR) | PR существует |
| Ваша работа получила 2 рецензии | Бот ставит лейбл `peer-review-complete` |
| Вы сами проверили 2 чужие работы | Счётчик в state |
| Каждая ваша рецензия: formal review (Approve / Request changes) | GitHub API |
| Каждая ваша рецензия: минимум 2 inline-комментария | GitHub API |

Когда все условия выполнены, бот поставит лейбл **`peer-review-complete`** на ваш PR и напишет комментарий.

## 6. Типичные ошибки

**Забыл создать ветку и работаю в main**

```bash
# Создайте ветку сейчас — изменения перенесутся автоматически
git checkout -b alice/hw2
git push origin alice/hw2
```

**Запушил в main напрямую**

Это заблокировано настройками репозитория — пуш не пройдёт. Создайте ветку (см. выше).

**Неправильное имя ветки**

Бот отклонит PR с комментарием. Переименуйте ветку:
```bash
git branch -m alice/hw2                    # переименовать локально
git push origin alice/hw2                  # запушить с правильным именем
git push origin --delete неправильное-имя  # удалить старую ветку
```

**Файл лежит не в той папке**

Файл должен быть в `reviews/{ваш_логин}/theory.md`. Переместите и перезапушьте:
```bash
mkdir -p reviews/alice
git mv reviews/чужой_логин/theory.md reviews/alice/theory.md
git commit -m "fix: correct file path"
git push origin alice/hw2
```

**Создан форк вместо клонирования**

Удалите форк на GitHub, клонируйте оригинальный репозиторий напрямую (см. Шаг 1).

**Неправильный заголовок PR**

Отредактируйте заголовок прямо в интерфейсе GitHub: откройте PR → нажмите карандаш рядом с заголовком → исправьте на `hw2: alice` → сохраните.

## 7. Частые вопросы

**Я открыл PR, но рецензент не назначен — почему?**

Вы сдали первым или одним из первых. Рецензент будет назначен автоматически когда следующий студент откроет свой PR. Ждать ничего не нужно.

**Рецензент попросил исправления (Request changes) — что делать?**

Внесите правки в `reviews/{ваш_логин}/theory.md`, запушьте в ту же ветку:
```bash
git add reviews/alice/theory.md
git commit -m "hw2: fix theory after review"
git push origin alice/hw2
```
PR обновится автоматически, рецензент получит уведомление.

**Прошло 3 дня, а рецензент молчит — что делать?**

Бот автоматически напомнит рецензенту. Если через ещё 2 дня и ничего не изменилось — напишите преподавателю.

**Для следующего ДЗ нужно клонировать репозиторий заново?**

Нет. Перейдите в папку `peer-review-hub`, обновите main и создайте новую ветку:
```bash
cd peer-review-hub
git checkout main
git pull origin main
git checkout -b alice/hw3
```
